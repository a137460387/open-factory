# Design Goals

## 核心目标

open-factory 的目标是提供一个可审计、可本地运行的桌面视频字幕编辑器。用户可以在本机导入媒体、编辑时间线、生成或导入字幕、预览项目、保存工程文件，并通过本地 FFmpeg 导出成片。

项目优先保证基础剪辑流程可靠：媒体导入、时间线编辑、字幕处理、预览、工程保存与恢复、Relink、缓存、代理媒体和导出队列都应能在没有账号和远程渲染服务的情况下工作。字幕翻译属于可选联网能力，必须由用户主动启用并确认第三方服务条款。

## 架构原则

- 本地优先：媒体文件、工程文件、缓存和导出结果默认保留在用户设备上。
- 清晰边界：纯时间线算法和导出规划放在 `packages/editor-core`，桌面 UI 和 Tauri 系统能力分别保留在各自层内。
- 可测试：核心时间线、项目迁移、缓存键、Relink、导出规划和队列状态需要单元测试覆盖。
- 可恢复：项目文件应尽量保持向前迁移能力，缺失媒体不应阻塞项目打开。
- 可取消：长耗时任务应异步执行，导出和后台媒体任务需要明确状态和取消路径。
- 最小外部依赖：网络能力必须可辨识、可关闭，并避免上传用户媒体。

## 技术选型理由

- Tauri 2 提供较小的桌面壳、系统文件选择器、进程调用和本地路径能力，适合本地媒体工具。
- React 和 TypeScript 提供可维护的 UI 组件模型和类型约束，便于维护复杂编辑状态。
- Zustand 用于轻量状态管理，命令对象负责时间线变更，避免 React 组件直接改写核心时间线结构。
- `editor-core` 保持纯 TypeScript，便于在单元测试中验证时间线算法、项目迁移和 FFmpeg 参数规划。
- FFmpeg 作为本地导出引擎，能力成熟且可通过参数数组调用，避免 shell 字符串执行风险。
- Playwright 和 Vitest 分别覆盖用户流程与核心算法，配合 Tauri smoke 测试验证关键原生能力。
