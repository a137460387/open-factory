## 三、性能与稳定性审计 ⚠️

**总体风险等级**: MEDIUM

### 3.1 HIGH 级别问题（需立即修复）

#### HIGH-1: 时间片超时被误判为任务失败 ⚡

**文件**: `packages/editor-core/src/core/task-scheduler.ts:304-321`

```typescript
private async executeWithTimeSlice<T>(
  execute: () => Promise<T>,
  timeSliceMs: number,
): Promise<T> {
  const result = await Promise.race([
    execute(),
    new Promise<never>((_, reject) => {
      setTimeout(() => {
        reject(new Error('TIME_SLICE_EXCEEDED'));
      }, timeSliceMs);
    }),
  ]);
  return result;
}
```

**问题**: `TIME_SLICE_EXCEEDED` 错误进入 catch 块并计入重试次数。默认 `timeSliceMs` 为 16ms，任何超过 16ms 的任务都会被错误地标记为失败并重试，最多重试 3 次后标记为 `failed`。一个 50ms 的任务会被错误地执行 4 次。

**影响**: 任务被无效重复执行，浪费计算资源，可能导致 UI 卡顿。

**修复建议**: 在 catch 块中识别 `TIME_SLICE_EXCEEDED` 错误，将其视为调度信号而非任务失败，不计入重试次数。

---

#### HIGH-2: 帧缓存全量排序性能瓶颈 ⚡

**文件**: `packages/editor-core/src/render-cache.ts:146-155`

```typescript
private pruneToBudget(): void {
  if (this.bytes <= this.maxBytes) return;
  const entries = [...this.entries.values()];
  entries.sort((a, b) => a.ts - b.ts);
  for (const entry of entries) {
    if (this.bytes <= this.maxBytes) break;
    this.delete(entry.key);
  }
}
```

**问题**: `pruneToBudget` 在每次 `put` 和 `retainAround` 时调用。如果缓存接近上限，每次写入都会触发 O(n log n) 全量排序。在高频帧缓存写入场景下（如快速播放），这可能成为性能瓶颈。

**影响**: 高频帧写入时可能导致帧丢失。

**修复建议**: 改用 MinHeap 或维护插入顺序的链表，避免每次全量排序。

---

#### HIGH-3: 抢占式调度可能导致任务 starvation ⚡

**文件**: `packages/editor-core/src/core/task-scheduler.ts:254-269`

```typescript
if (this.config.enablePreemption && entry.task.canInterrupt) {
  const higherPriorityTask = this.findHigherPriorityTask(entry.task.priority);
  if (higherPriorityTask) {
    entry.status = 'paused';
    if (entry.task.resumeAfterInterrupt) {
      await entry.task.resumeAfterInterrupt();
    }
    this.running.delete(entry.task.id);
    this.queue.set(entry.task.id, entry);
    this.schedule();
    return;
  }
}
```

**问题**: 被抢占的任务被重新放回队列，但 `starvationPrevention` 机制只在任务等待超过 10 秒时才将评分提升到 `normal` 级别。如果 `normal` 级别任务持续涌入，`background` 任务仍然会被饿死。

**影响**: 低优先级任务可能永远无法完成。

**修复建议**: 被抢占的任务应获得临时优先级提升，或设置最大抢占次数限制。
