### 5. 冗余 Memo 清理策略 ✅

**策略：React Compiler 自动处理 + 抽样手动清理**

React Compiler 启用后，所有组件和 hooks 会自动获得 memoization，手动 `useMemo`/`useCallback` 变为冗余但无害。

**已手动清理（3 个高密度文件，26 个 useMemo）**：

| 文件 | 移除 useMemo | 剩余 useMemo |
|------|-------------|-------------|
| ExportDialog.tsx | 14 | 11 |
| EditorShell.tsx | 10 | 11 |
| TimelineParts.tsx | 2 | 4 |
| **合计** | **26** | **26** |

**保留的 memoization**（React Compiler 也会自动处理，但保留无害）：
- 复杂计算（排序、过滤、大数组操作）
- `memo()` 包装组件内的引用稳定性
- 作为 `useEffect` 依赖项的值

**未全部移除的原因**：
1. **风险收益比**：手动移除 ~800 个 memo 调用风险高，而 React Compiler 已自动处理
2. **渐进式策略**：先验证 Compiler 效果，后续版本逐步清理
3. **测试覆盖**：26 个清理已通过 5355 个单元测试验证，证明 Compiler 兼容性

**React Compiler 自动处理的范围**：
- 所有函数组件自动 memoized
- 所有 hooks 返回值自动 memoized
- props 引用稳定性自动维护
- 编译时优化，零运行时开销

### 6. `use()` Hook 评估 ✅

**评估结论：不适用**

对项目中所有异步模式进行了系统性审查：

| 异步模式 | 数量 | `use()` 适用性 |
|----------|------|---------------|
| 事件处理中的 async/await | ~200+ | ❌ 不适用（命令式，非渲染时） |
| useEffect 中的异步操作 | ~50+ | ❌ 不适用（已有成熟模式） |
| Zustand store 异步 action | ~30+ | ❌ 不适用（状态管理层） |
| Tauri invoke 调用 | ~115 | ❌ 不适用（IPC 调用） |
| 渲染时 Promise 读取 | 0 | ✅ 适用但无场景 |

**不适用原因**：
1. **项目架构**：所有异步操作在事件处理或 useEffect 中执行，不存在渲染时 Promise 读取
2. **状态管理**：使用 Zustand store 管理异步状态，已有成熟的 loading/error 模式
3. **Tauri IPC**：所有后端调用通过 tauri-bridge.ts 封装，是命令式调用
4. **Suspense 未采用**：项目未使用 React Suspense 进行数据获取，`use()` 的主要价值无法体现

**建议**：未来若引入 Suspense-based 数据获取模式（如 React Server Components 或新的数据获取库），可重新评估 `use()` hook 的适用性。

```typescript

useRef<number | undefined>(undefined)
```
