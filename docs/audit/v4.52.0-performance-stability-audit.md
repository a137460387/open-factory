# v4.52.0 性能与稳定性审计报告

**审计日期**: 2026-07-21
**审计范围**: 内存池、帧缓存 LRU、任务调度器、WebGPU/WebGL2 双后端、Worker 间通信
**总体风险等级**: MEDIUM

---

## 执行摘要

v4.52.0 引入了内存池、帧缓存 LRU、任务调度器等底层性能优化。审计发现 **0 个 CRITICAL**、**3 个 HIGH**、**5 个 MEDIUM**、**4 个 LOW** 级别问题。核心机制设计合理，但存在若干稳定性风险需要关注。

## 1. 内存池机制审查

**文件**: `packages/editor-core/src/core/memory-pool.ts`

### 1.1 实现分析

`MemoryPool` 采用 Map 存储池化对象，支持 LRU 淘汰（`evictLRU`）和自动 GC（`runGC`）。GC 使用 `setTimeout` + `requestIdleCallback` 调度，目标将内存降至 70% 使用率。

**优点**:
- LRU 淘汰基于 `lastAccessedAt` 线性扫描，逻辑正确（第 268-290 行）
- GC 评分算法综合考虑访问频率和最近访问时间（第 219-222 行）
- Transferable Objects 支持完善，`ImageBitmap.close()` 正确调用（第 319-327 行）

### 1.2 发现的问题

#### [MEDIUM-1] GC 排序算法存在 O(n log n) 开销

**文件**: `packages/editor-core/src/core/memory-pool.ts:218-223`

```typescript
const entries = [...this.objects.values()];
entries.sort((a, b) => {
  const scoreA = a.accessCount / (Date.now() - a.lastAccessedAt + 1);
  const scoreB = b.accessCount / (Date.now() - b.lastAccessedAt + 1);
  return scoreA - scoreB;
});
```

当池中有 1000 个对象时，每次 GC 都会创建数组副本并排序。在 60fps 渲染循环中触发 GC 可能造成帧丢失。

**建议**: 维护一个按访问分数排序的辅助数据结构（如跳表），避免每次 GC 全量排序。

#### [MEDIUM-2] `requestIdleCallback` 递归 GC 无深度限制

**文件**: `packages/editor-core/src/core/memory-pool.ts:242-247`

```typescript
if (this.totalBytes > targetBytes && typeof requestIdleCallback !== 'undefined') {
  this.idleCallbackId = requestIdleCallback(() => {
    this.idleCallbackId = null;
    this.runGC();
  });
}
```

如果持续有新对象进入池中，`runGC` 可能无限递归调用自身。虽然 `requestIdleCallback` 在空闲时执行，但在极端情况下可能导致持续的 GC 压力。

**建议**: 增加递归深度计数器或最大重试次数。

#### [LOW-1] `evictLRU` 线性扫描效率

**文件**: `packages/editor-core/src/core/memory-pool.ts:268-290`

每次 `acquire` 空间不足时调用 `evictLRU`，该方法遍历所有对象找最旧条目。在热路径上调用时可能影响性能。

**建议**: 维护一个有序链表或使用 LRU Cache 的经典双向链表实现。

## 2. 帧缓存 LRU 机制审查

**文件**: `packages/editor-core/src/render-cache.ts` (TimelineRenderFrameCache)
**文件**: `apps/desktop/src/lib/preview/gpu-acceleration.ts` (GpuTexturePool)

### 2.1 TimelineRenderFrameCache 分析

`TimelineRenderFrameCache` 使用 Map 存储帧缓存条目，按时间戳排序淘汰。

**优点**:
- `retainAround` 方法基于播放头位置智能保留附近帧（第 99-112 行）
- `disposeBitmap` 回调确保 `ImageBitmap` 被正确 `close()`（第 157-165 行）
- 字节级内存预算控制（`maxBytes` 默认 256MB）

#### [HIGH-1] `pruneToBudget` 每次调用都创建数组副本并排序

**文件**: `packages/editor-core/src/render-cache.ts:146-155`

```typescript
private pruneToBudget(): void {
  if (this.bytes <= this.maxBytes) return;
  const entries = [...this.entries.values()];
  entries.sort((a, b) => a.ts - b.ts);
  for (const entry of entries) {
    if (this.bytes <= this.maxBytes) break;
    this.delete(entry.key);
  }
}
```

`pruneToBudget` 在每次 `put` 和 `retainAround` 时调用。如果缓存接近上限，每次写入都会触发全量排序。在高频帧缓存写入场景下（如快速播放），这可能成为性能瓶颈。

**建议**: 改用 MinHeap 或维护插入顺序的链表，避免每次全量排序。

### 2.2 GpuTexturePool 分析

**文件**: `apps/desktop/src/lib/preview/gpu-acceleration.ts:38-141`

`GpuTexturePool` 使用单调递增的 `clock` 计数器实现 LRU（第 63 行），淘汰时线性扫描找最小 `lastUsed` 值（第 127-140 行）。

#### [MEDIUM-3] GpuTexturePool 淘汰效率与 TimelineRenderFrameCache 相同问题

**文件**: `apps/desktop/src/lib/preview/gpu-acceleration.ts:127-140`

```typescript
private pruneToBudget(): void {
  while (this.usedBytes > this.maxBytes && this.entries.size > 0) {
    let oldest: GpuTexturePoolEntry<TTexture> | undefined;
    for (const entry of this.entries.values()) {
      if (!oldest || entry.lastUsed < oldest.lastUsed) {
        oldest = entry;
      }
    }
    if (!oldest) break;
    this.delete(oldest.key);
  }
}
```

最坏情况下（全部淘汰），复杂度为 O(n^2)。512MB 上限下可能存储大量纹理。

**建议**: 同 HIGH-1，改用有序数据结构。

### 2.3 GPUColorProcessor 缓存分析

**文件**: `packages/editor-core/src/color/gpu-color-processing.ts:1177-1199`

#### [MEDIUM-4] GPUColorProcessor 缓存无内存限制，仅限制条目数

**文件**: `packages/editor-core/src/color/gpu-color-processing.ts:1179`

```typescript
if (this.cache.size >= this.config.maxCacheSize) {
  // 淘汰最旧的
}
```

缓存仅按条目数限制（默认 64），每个条目存储 `Uint8ClampedArray` 帧数据。对于 4K 分辨率（3840x2160x4 = 33MB/帧），64 帧缓存可达 **2.1GB**，远超合理内存预算。

**建议**: 改用字节级内存限制，与 `TimelineRenderFrameCache` 保持一致。

#### [LOW-2] GPUColorProcessor.processFrame 的 GPU 路径实际走 CPU

**文件**: `packages/editor-core/src/color/gpu-color-processing.ts:1095-1107`

```typescript
} else {
  // GPU 处理路径 - 在实际 GPU 上下文中由调用者管理
  // 这里回退到 CPU 以保持纯逻辑层
  output = cpuProcessFrame(...);
}
```

`processFrame` 声称支持 GPU 加速，但当 `currentBackend` 为 `webgl2` 或 `webgpu` 时，实际仍走 CPU 回退路径。这是一个误导性的 API 设计。

**建议**: 要么实现真正的 GPU 处理路径，要么在文档/类型中明确标注此方法始终使用 CPU。

## 3. 任务调度器审查

**文件**: `packages/editor-core/src/core/task-scheduler.ts`

### 3.1 实现分析

`TaskScheduler` 实现了基于优先级的抢占式任务队列，支持 5 级优先级（immediate/high/normal/low/background）和优先级老化机制。

### 3.2 发现的问题

#### [HIGH-2] 时间片实现存在任务丢失风险

**文件**: `packages/editor-core/src/core/task-scheduler.ts:304-321`

```typescript
private async executeWithTimeSlice<T>(
  execute: () => Promise<T>,
  timeSliceMs: number,
): Promise<T> {
  const result = await Promise.race([
    execute(),
    new Promise<never>((_, reject) => {
      setTimeout(() => {
        reject(new Error('TIME_SLICE_EXCEEDED'));
      }, timeSliceMs);
    }),
  ]);
  return result;
}
```

时间片超时通过 `reject` 抛出 `'TIME_SLICE_EXCEEDED'` 错误。在 `executeTask` 的 catch 块中（第 285-301 行），此错误会被视为任务失败并计入重试次数。但 `TIME_SLICE_EXCEEDED` 并非真正的任务失败，而是调度信号。

当 `timeSliceMs` 设置为 16ms（默认值）时，任何超过 16ms 的任务都会被错误地标记为失败并重试，最多重试 3 次后标记为 `failed`。这意味着一个 50ms 的任务会被错误地执行 4 次（1 次 + 3 次重试）。

**建议**: 在 catch 块中识别 `TIME_SLICE_EXCEEDED` 错误，将其视为调度信号而非任务失败，不计入重试次数。

#### [HIGH-3] 抢占式调度可能导致任务 starvation

**文件**: `packages/editor-core/src/core/task-scheduler.ts:254-269`

```typescript
if (this.config.enablePreemption && entry.task.canInterrupt) {
  const higherPriorityTask = this.findHigherPriorityTask(entry.task.priority);
  if (higherPriorityTask) {
    entry.status = 'paused';
    if (entry.task.resumeAfterInterrupt) {
      await entry.task.resumeAfterInterrupt();
    }
    this.running.delete(entry.task.id);
    this.queue.set(entry.task.id, entry);
    this.schedule();
    return;
  }
}
```

被抢占的任务被重新放回队列头部（`this.queue.set`），但 `selectNextTask` 仍然基于优先级评分选择。如果持续有高优先级任务进入队列，低优先级任务可能永远无法完成。

虽然有 `starvationPrevention` 机制（第 232-238 行），但它只在任务等待超过 `starvationPreventionMs`（默认 10 秒）时才将评分提升到 `normal` 级别。如果 `normal` 级别任务持续涌入，`background` 任务仍然会被饿死。

**建议**: 被抢占的任务应获得临时优先级提升，或设置最大抢占次数限制。

#### [MEDIUM-5] `schedule` 方法的重入保护过于简单

**文件**: `packages/editor-core/src/core/task-scheduler.ts:190-209`

```typescript
private async schedule(): Promise<void> {
  if (this.scheduling) return;
  this.scheduling = true;
  // ...
  this.scheduling = false;
}
```

`scheduling` 标志在 `schedule` 返回时重置为 `false`。但由于 `schedule` 是异步方法，在 `await` 点之间可能有新的 `schedule` 调用被跳过。这在高频任务提交场景下可能导致调度延迟。

**建议**: 改用调度队列模式，将待调度请求排队并在当前调度完成后处理。

#### [LOW-3] completed Map 无上限增长

**文件**: `packages/editor-core/src/core/task-scheduler.ts:74`

```typescript
private completed: Map<string, TaskEntry> = new Map();
```

已完成的任务条目永远不会被清理，长期运行的应用中会导致内存持续增长。

**建议**: 增加 `maxCompletedEntries` 配置，或提供 `clearCompleted()` 方法。

## 4. WebGPU/WebGL2 双后端审查

### 4.1 GPU 后端检测

**文件**: `packages/editor-core/src/color/gpu-color-processing.ts:963-1041`

`GPUColorProcessor.initialize()` 实现了 WebGPU -> WebGL2 -> CPU 的三级回退策略。

**优点**:
- 回退链逻辑清晰（第 968-1041 行）
- WebGL2 上下文创建带有 `WEBGL_debug_renderer_info` 扩展检测（第 1005 行）
- 状态通知机制完善（`onStatusChange` 回调）

#### [LOW-4] WebGPU adapter/device 获取后未保存引用

**文件**: `packages/editor-core/src/color/gpu-color-processing.ts:971-991`

```typescript
const adapter = await navigator.gpu.requestAdapter();
if (adapter) {
  const device = await adapter.requestDevice();
  if (device) {
    this.currentBackend = 'webgpu';
    this.deviceInfo = { ... };
```

WebGPU `adapter` 和 `device` 获取成功后，仅设置了 `currentBackend` 和 `deviceInfo`，但未保存 `device` 引用。后续无法使用该设备进行实际 GPU 计算。

**建议**: 保存 `device` 引用以供后续 GPU 计算使用。

### 4.2 WebGL 合成器资源管理

**文件**: `apps/desktop/src/lib/preview/webgl-compositor.ts`

#### [MEDIUM-6] drawSourceWithColorNodeGraph 创建临时 WebGL 上下文未复用

**文件**: `apps/desktop/src/lib/preview/webgl-compositor.ts:373-395`

```typescript

scratch.width = width;
scratch.height = height;
```

**建议**: 缓存 scratch canvas 和 compositor 实例，复用 WebGL 上下文。

### 4.3 转场渲染器资源管理

### 5.2 发现的问题

#### [MEDIUM-7] color-scopes Worker 接收 Uint8Array 帧数据未使用 Transferable

**文件**: `apps/desktop/src/workers/color-scopes.worker.ts:11-16`

```typescript
self.onmessage = (event: MessageEvent<ColorScopesWorkerRequest>) => {
  try {
    const { width, height, data, waveformColumns } = event.data;
```

`data` 字段为 `Uint8Array` 类型的帧像素数据（RGBA），对于 1080p 帧约为 8MB。每次 postMessage 都会进行结构化克隆（深拷贝），造成约 8MB 的内存分配和数据复制。

**建议**: 在主线程发送时使用 `{ transfer: [data.buffer] }`，或在 Worker 内使用 `OffscreenCanvas` 直接处理。

#### [MEDIUM-8] ai-worker 的 inputData 使用 Float32Array 未标记 Transferable

**文件**: `packages/editor-core/src/ai/ai-worker.ts:271-277`

```typescript
slot.worker.postMessage({
  id: request.id,
  type: request.type,
  modelType: request.modelType,
  inputData: request.inputData,
  config: request.config,
} as AIWorkerRequest);
```

`inputData` 类型为 `Float32Array | number[]`。对于大型张量（如 1024x1024 float32 = 4MB），结构化克隆开销显著。

**建议**: 当 `inputData` 为 `Float32Array` 时，使用 `{ transfer: [inputData.buffer] }` 传递。

#### [LOW-5] Worker 创建模式：每个任务创建新 Worker

**文件**: `apps/desktop/src/media/waveform.ts:147-168`

```typescript
function extractWithWorker(...): Promise<WaveformResult> {
  return new Promise((resolve, reject) => {
    const worker = new Worker(new URL('../workers/waveform.worker.ts', import.meta.url), { type: 'module' });
    worker.onmessage = (event) => {
      worker.terminate();
      // ...
    };
  });
}
```

waveform Worker 每次提取波形都创建新 Worker 实例，任务完成后立即 terminate。Worker 创建开销（解析 JS、初始化上下文）约为 5-20ms，在批量处理多个音频片段时会累积。

**建议**: 改用 Worker 池模式复用 Worker 实例，类似 `AIWorkerPool` 的设计。
