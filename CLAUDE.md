## 审计类任务的防失真规范

在执行代码审计、死代码扫描、依赖分析等"生成清单类"任务时，必须遵守以下规则，防止结论失真：

### 1. 原始工具输出必须落盘，禁止转述后归档

任何基于工具输出（knip、cargo audit、eslint、CI 日志等）生成的审计文档，**不得**只写"汇总结论"，必须同时：

- 将工具的完整原始输出保存为独立文件（如 `docs/evidence/<tool>-<date>.txt`）
- 审计文档中的每一条结论，附带指向原始文件的行号引用，而不是靠记忆/复述转写
- 如果无法保存完整原始输出（如日志过长），至少保存关键片段的原文摘录，并注明摘录来源命令

**反面案例**：曾出现审计文档声称 10 个类型分布在 3 个不存在的文件路径中（`export/types.ts`、`transition.ts`、`render-queue/types.ts`），类型名和路径均为凭空捏造，事后查证 git 全历史中从未存在过这些文件。根因是工具输出被"整理成表格"时脱离了原始数据，变成了基于印象的重写。

### 2. 删除/修改前必须独立验证目标存在

对审计清单中每一项，执行操作前必须先独立确认：

- 目标（函数名/类型名/文件路径）在当前代码库中**确实存在**（grep 或读文件确认，不能信任清单本身）
- 目标在原始工具输出中**确实被报告过**（比对原始文件，不能只信任二次汇总的清单）
- 存在 ≠ 检测来源可信：一个真实存在的函数名，如果在原始工具输出里查无此项，说明它是被"凑巧编对"还是真的被检测到，需要明确标注，不能默认为可信

### 3. 数字必须一致，任何对不上都要查清楚再继续

执行过程中如果出现前后不一致的数字或事实（如"3 个 commit" vs "4 个 commit"、"~60 行" vs "106763 行"、"commit sha 对不上 run id"），**必须先停下核实根因**，不能因为"结果好像没问题"就跳过。这类矛盾往往是系统性问题的早期信号，而不是无关紧要的笔误——即便最终查实只是笔误，核实过程本身也是必要的止损动作。

### 4. 危险操作前用只读命令重新核实状态

长会话（尤其接近 context 上限时）可能产生与当前任务无关的幻觉输出。执行任何写操作（git commit / push / merge、文件删除、依赖变更）前，如果:

- 会话已运行很长时间，或
- 输出内容与当前任务明显不符（提到了不相关的项目、文件、功能）

应先用纯只读命令重新核实真实状态（`git status`、`git log`、`git diff`、`ls`/`Glob` 确认文件存在性），确认无污染后再执行写操作。不要信任"已完成"的自我汇报，除非有可验证的原始输出支撑。

### 5. 范围必须锁定，不擅自扩大或跳过验证

- 只执行经过确认批准的清单项，不要顺手处理清单外的"顺便发现的"问题
- 每一类/每一批操作后必须先验证（typecheck / test / build 等）通过，再进入下一批，不能因为"看起来没问题"就连续跳过多步
- 收到暂停指令后，必须停在当前最小操作单元，不能预判会被批准就继续往后跑
